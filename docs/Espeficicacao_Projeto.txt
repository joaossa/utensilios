# 🧩 1. DESCRIÇÃO DO PROJETO

## 🎯 Objetivo

Desenvolver um sistema web para **gestão de itens e empréstimos de uma igreja**, permitindo controle completo de:

* Inventário de itens (equipamentos, mobiliário, etc.)
* Cadastro de membros autorizados
* Empréstimos, devoluções e atrasos
* Histórico e rastreabilidade

📌 A essência do sistema é:
👉 **"Saber o que temos, quem pegou, quando devolve e em que estado está"** 

---

## 🧱 Escopo funcional (visão macro)

### Módulos principais:

1. **Cadastro de Itens**
2. **Cadastro de Membros**
3. **Empréstimos e Devoluções**
4. **Histórico e Auditoria**
5. **Relatórios e Alertas**

📌 Esses três primeiros módulos já formam o **MVP funcional completo** 

---

## 💡 Diferencial estratégico (minha opinião)

Esse sistema tem um potencial enorme se você incluir desde o início:

* Controle visual (fotos)
* Histórico completo (auditável)
* Status inteligente (ativo, atrasado, devolvido)
* Base preparada para crescimento (ex: reservas futuras)

👉 Isso evita retrabalho depois.

# 🧱 2. DEFINIÇÃO DAS ENTIDADES

Agora vamos estruturar isso no nível de banco/modelagem (DDD leve).

---

## 📦 2.1 Entidade: ITEM

Representa o objeto físico da igreja.

### Campos principais:

* id
* codigo (único)
* descricao
* categoria
* estado (novo, bom, manutenção, danificado)
* localizacao
* disponibilidade
* data_cadastro

📸 Relacionamento:

* **1:N com imagens (ItemImagem)**

📌 Observação importante:

* Múltiplas fotos por item são essenciais 

---

## 🖼️ 2.2 Entidade: ITEM_IMAGEM

* id
* item_id (FK)
* url_imagem
* descricao (opcional)
* ordem

---

## 👤 2.3 Entidade: MEMBRO

Representa quem pode pegar itens.

### Campos:

* id
* nome
* telefone
* email
* tipo (membro, liderança, visitante autorizado)
* ativo (S/N)
* data_cadastro

📌 Pode evoluir depois para:

* controle de permissões
* restrições de empréstimo

---

## 🔄 2.4 Entidade: EMPRESTIMO

Coração do sistema.

### Campos:

* id
* item_id (FK)
* membro_id (FK)
* data_retirada
* data_prevista_devolucao
* data_devolucao (nullable)
* status (ativo, devolvido, atrasado)
* observacoes

📌 Estrutura exatamente alinhada ao que você definiu 

---

## 📜 2.5 Entidade: HISTORICO_ITEM (RECOMENDADO 🔥)

Essa é evolução profissional do sistema.

* id
* item_id
* tipo_evento (emprestado, devolvido, manutenção)
* descricao
* data_evento
* usuario_responsavel

📌 Isso garante:
👉 Auditoria
👉 Transparência
👉 Base para relatórios

---

## 🔔 2.6 (OPCIONAL FUTURO) NOTIFICACOES

* lembrete de devolução
* atraso
