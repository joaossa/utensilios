5. Variaveis de ambiente realmente relevantes

Frontend:
- obrigatoria:
  - `VITE_API_BASE_URL=https://api.ibgutensilios.com.br`
- regra oficial do Vite:
  - so variaveis com prefixo `VITE_` chegam ao cliente
  - portanto segredo nunca entra em `VITE_*`

API - obrigatorias no codigo atual:
- `DATABASE_URL`
- `JWT_SECRET`

API - obrigatoria por seguranca operacional:
- `CORS_ORIGINS`
  - usar pelo menos:
    - `https://ibgutensilios.com.br,https://www.ibgutensilios.com.br`
  - se o dominio da API tambem precisar ser aceito em alguma frente publica futura, adicionar de forma explicita

API - opcional/controlada pelo ambiente:
- `PORT`
  - o codigo usa `process.env.PORT ?? 3001`
  - conclusao pratica:
    - se a Hostinger injetar `PORT`, o codigo ja respeita
    - `3001` hoje e apenas fallback do codigo e referencia local
    - nao endurecer o deploy em torno de porta fixa sem erro real da conta

API - usadas apenas em seed controlado:
- `SEED_ADMIN_EMAIL`
- `SEED_ADMIN_PASSWORD`
- `SEED_INCLUDE_SAMPLE_DATA`

Variaveis citadas em documentos antigos, mas nao localizadas no codigo atual:
- `WEB_BASE_URL`
- `RECOVERY_PASSWORD_URL`

Conclusao sobre essas duas variaveis:
- hoje elas nao devem bloquear o deploy
- se uma frente futura de recuperacao de senha for aberta, elas podem voltar a ser necessarias

6. Alertas importantes encontrados nesta releitura

Alerta 1 - divergencia entre documentacao antiga e codigo atual da API:
- alguns documentos antigos ainda falam como se o painel fosse apontar diretamente para `dist\src\server.js`
- o codigo atual, os artefatos e o pacote da API ja foram adaptados para `index.js` como entrada mais compativel
- a partir desta data, este arquivo passa a ser a referencia principal para esse ponto

Alerta 2 - `validarDev.ps1` nao recompila mais TypeScript da API:
- o script ainda escreve `Compilando a API`
- porem ele chama `npm run build`
- hoje `npm run build` na API executa apenas `node hostinger-stage.js build`
- a compilacao TypeScript real esta em `npm run build:ts`
- implicacao:
  - `validarDev.ps1` continua bom para:
    - validar isolamento dos `.env` locais
    - rodar migrations locais
    - subir `dist\src\server.js`
    - validar `/health`
    - validar `vite preview`
  - mas ele nao prova sozinho uma nova transpilacao da API
- proxima acao tecnica recomendada:
  - alinhar `validarDev.ps1` com `npm run build:ts` antes de trata-lo como gate unico de compilacao da API

Alerta 3 - CORS vazio libera qualquer origem:
- pelo comportamento atual de `src\app.ts`, producao nao pode subir com `CORS_ORIGINS` vazio

Alerta 4 - DDL exportado nao e a melhor fonte de verdade:
- `docs\scripts_banco\Banco_Estrutura_DDL_utensilios_v2.txt` ajuda como referencia historica
- porem o estado estrutural mais confiavel hoje esta nas migrations e seeds atuais

Alerta 5 - documentacao oficial da Hostinger precisa ser lida no produto certo:
- existem paginas antigas e genericas de hospedagem Node
- para este projeto, a linha correta e a de `Node.js Web Apps`
- em caso de conflito, prevalecem:
  - o codigo real do repositorio
